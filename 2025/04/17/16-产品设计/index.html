16-产品设计

LLM
Publish Date:   2025-04-17

1 产品设计

简单说，就三件事：

  1. 功能设计 - 干什么
  2. 信息架构设计 - 怎么摆放这些功能
  3. 交互设计 - 怎么让用户用起来顺手

如果对视觉要求高，还需要特别做视觉设计。否则套模板就差不多。

对全栈来说，三件事不分先后，而是彼此交错。

但起手做的，常常是：借鉴。

2 借鉴产品

借鉴的合理性：

  1. 不该打破用户习惯
  2. 这其实是行业惯例了

优雅的姿势：

  1. 先自己设计，然后再参考 - 有独立的想法，不会盲目抄
  2. 学结构、学交互，别学视觉 - 「像素级」抄袭是会被嘲笑的
  3. 取各家之长做整合、抽象 - 有机会抛弃别人的桎梏，另起炉灶

避坑关键：

  1. 用 5 why 法推演为什么人家这么设计
  2. 如果找不到理性答案，慎抄，容易进坑

3 写 PRD

一定要先写文档！俗称 PRD（Product Requirement Document）。

  1. 虽然可能写完没人看，但写文档的过程比结果更重要，可以让思路更清晰
  2. 飞书的「产品需求文档」模板特别好用，往里填就行
  3. 可以用 ChatGPT 找些思路和灵感，常见功能它写得还不错
  4. 试试在 IDE 里开着 Copilot 写文档，体验很棒（「体验很棒」这四个字，就是 Copilot 写的）

4 AI 产品设计原则

在设计产品的过程中，时不时检查，是否这些原则全部符合。

4.1 GitHub Copilot 总结的四个关键支柱

GitHub Copilot 是最早，也是迄今为止最成功的基于大模型 AI 的垂类工具产品。其经验非常值得借鉴。

Copilot 团队也乐于分享。在 How we’re experimenting with LLMs to evolve GitHub Copilot 一文中，总结了 GitHub 设计 AI 产品的四个关键支柱（Key Pillars）：

  • 可预测。我们希望创建能引导开发人员实现最终目标的工具，但不会让他们感到惊讶或不知所措。
  • 可容错。正如我们所见，AI 模型可能犯错。用户应该能够轻松地发现不正确的建议，并以较低的成本来解决，以便专注和提高生产力。
  • 可操控。当回答不正确或不符合用户期望时，他们应该能够引导 AI 找到解决方案。否则，我们就是乐观地指望模型产生完美的答案。
  • 可验证。解决方案必须易于评估。模型并不完美，但如果用户能验证其输出，它们就能成为非常有用的工具。

4.2 七个实操经验

以下内容综合了我们教研团队的实战经验、有赞 CEO 白鸦的分享和微软的 Collabrative UX 实践。

  1. 永远不要让 AI 直接生效
    1. 人工确认是必须的
    2. AI 幻觉永远存在
  2. 代替操作和探索，不要完全代替思考
    1. 把 AI 当人看，把用户当老板
    2. 不思考的老板是不配发号施令的
    3. 让人思考该输入什么，和思考 AI 输出是否正确
  3. 尽量减少 chat
    1. 先用表单。准确、快捷
    2. 对话是兜底方案。选项解决不了，或实在太多时才用
  4. 直接给结果
    1. 用户只想要结果，没耐心看太多
    2. 如果 agent、思维链、工具调用等过程太耗时，报进度
  5. 明确告知这是 AI 的输出
    1. 给用户可能有错的预期，起到一定免责效果
    2. 不要欺骗，不要欺骗，不要欺骗
  6. 收集反馈
    1. 有反馈才有优化
    2. 让用户主动上报，比如 ChatGPT 的 👍 和 👎
    3. 采集用户行为，比如 GitHub Copilot 里是否按了 tab 键，ChatGPT 里是否点击了 copy 按钮
  7. 尊重传统
    1. AI 不是颠覆，是增强
    2. 传统产品设计经验，全都继续有效

4.3 UE/UX 设计经验

https://www.shapeof.ai/ 总结了几大方面：

  • 识别 - 帮助用户识别和区分 AI 特性和内容
    • 色彩模式 - 提供视觉提示，帮助用户识别内容中的 AI 特性
    • 图标 - 用图像来表达产品中使用的 AI 形态
    • 命名 - 我们如何称呼 AI？
  • 向导 - 提供线索，帮助用户理解 AI 的工作原理以及如何与之协作
    • 引导 - 提醒用户可以在现有工具中使用 AI
    • 建议 - 向用户提供写提示词的线索
    • 模版 - 向用户提供结构化的模版，并且可以用 AI 预填写
  • 提示 - 用户用 AI 生成结果的不同方式
    • 自动填写 - 让用户轻松地将一个提示词扩展到多个输入
    • 开放对话 - 开放式提示词输入，从而能持续对话或调整结果
    • 混合 - 组合多个提示词或更多输入源，以获得新结果
    • 整理 - 将复杂的信息提炼或重组为简单的结构
  • 调节 - 细化提示或结果，以获得更可预测的输出
    • 过滤/参数 - 让用户定义约束以提高结果的质量
    • 模型管理 - 让用户指定提示词所使用的模型
    • 人格化语音 - 确保输出与您的声音、语调和偏好保持一致
  • 信任 - 让用户控制响应并评估其准确性
    • 警告 - 告知用户模型或技术整体上的不足或风险
    • 控制 - 管理信息流或在请求中途中止以调整提示词
    • 反馈 - 对模型与预期的差距或错误发出信号——但用户能分辨清楚吗？
    • 脚印 - 让用户能够追踪来源和结果之间的关系

5 原型设计

程序员会更喜欢直接上代码，省却画原型步骤。这个习惯不好：

  1. 代码写出来就有感情，不爱改
  2. 画原型图，能在具象和整体层面上，更好地把控。

和完整编制的团队不同：

  1. 独立开发画原型图不需要太精细，线框图足够
  2. 细节在代码里做调整，更直接、准确
  3. AI 直接生成原型图除外

5.1 AI 原型设计工具

用 prompt 出设计图。

测试 prompt：一个 AI 教学网站，叫「哎！师傅」。界面是对话风格，但是由 AI 主导对话，用户只能点击按钮选择下一步的动作。按钮可能是 1-4 个。

  1. Motiff（国际版）/妙多（中国版）
    1. AI 为核的全能设计工具
    2. 大模型出现之前立项开发，大模型之后迅速面向 AI 重构，2024 年 6 月正式发布
    3. AI 复制、AI 布局
    4. AI 识别系统组件，建立组件库，AI 做一致性检查
    5. AI 生成 UI
    6. AI 魔法框猜测想要的设计元素（很像 GitHub Copilot）
  2. Galileo AI
    1. 支持 Text to UI 和 Image to UI，设计精美
    2. 可以用 prompt 继续修改设计
    3. 可以手工调配主题色、字体等
    4. 可以直接导出到 Figma
  3. v0
    1. 支持 Text to UI 和 Image to UI
    2. 直接生成 React 前端代码，展示效果更真实
    3. 可以用 prompt 继续修改设计，细节控制优于 Galileo
  4. Figma
    1. 刚刚发布 AI 功能，和 Motiff 接近，但现在只能 Join the waitlist

建议流程：

  1. 纯用 Motiff
  2. 或者用 Galileo 出设计图，发给 v0，再用 v0 做细节调整

5.2 传统原型设计工具

推荐的工具：

  1. draw.io
    1. 免费开源
    2. 对程序员最友好，可以直接改 style
    3. 内置的图形库，几乎就足够用了
    4. 可以直接编辑云文件，包括 GitHub
    5. 保存的 .png 和 .svg 文件还能继续编辑
    6. 容器功能特别好用，画架构图首选
    7. 有客户端，速度更快
  2. Figma
    1. 最受欢迎的专业原型设计工具，非常非常强大，学习成本稍高
    2. 可以写 plugin 来扩展其功能
    3. 适合多人协作，支持实时协同编辑
    4. 直接生成样式代码
    5. 有客户端，速度更快
    6. 处理 logo、icon 等图片，免费版就够用。

6 极简 UI 设计原则

  1. 所有元素要对齐。这和保持代码缩进一样重要
  2. 亲密性，相关元素离得更近。和代码里加空行一个道理
  3. 同一界面最多使用 4 种颜色。颜色的意义：传递信息 > 美观
  4. 用 4 的倍数（4、8、12、16、20……）设置大小、间距。视觉感受不明显，但适配性更好

推荐《写给大家看的设计书（第 4 版）》。我用半本书就撑住了半辈子的设计工作……

7 快速搭 demo

可以有个期待，AI 把 PRD 直接变成产品。MetaGPT、Devin 在这方面走得最远，但还很不够。

懂代码的我们，可以用 Gradio，快速搭出一个主流程 demo，供用户/客户体验，验证需求。

https://www.gradio.app/

  2. 非常简单，用纯 python 就能写出功能强大的 Web 界面，不需要任何 JavaScript、CSS 知识
  3. 响应式设计，支持手机、平板、电脑等多种设备
  4. 文档详细，样例丰富，非常容易上手

7 迭代

划重点：谨慎增加新功能，保持小而美

  1. 别用高频迭代代表勤奋，要用积极运营代表勤奋
  2. 第一版只做主路径功能。但别忽略细节体验
  3. 用 KANO 模型，找到提升核心用户满意度的功能，只做这些
    1. 非核心用户骂你，也不要理睬
    2. 别轻易改变核心用户画像。如果要变，可能做个新产品更好

思考：兴奋型、期望型、基本型需求，怎样的优先级？
